# 社員管理アプリ 技術スタック・設計・フォルダ構成

## 1. この資料の目的

この資料は、講師が社員管理アプリ2026の技術構成、設計、処理の流れ、主要ファイルを把握するためのものです。

次の場面で利用します。

- 研修前の講師間の認識合わせ
- MVC・DB・APIの説明会
- 新入社員からの質問対応
- Pull Requestレビュー
- 課題追加やアプリ改修の影響範囲確認

記載内容は、現在のリポジトリに実装されているコードを基準としています。

## 2. アプリの概要

認証済み利用者が社員情報を管理する、社内研修用のLaravelアプリです。

### Web機能

- 利用者登録
- ログイン・ログアウト
- 社員一覧
- 社員登録・編集・論理削除
- 社員番号、氏名、部署による検索
- 部署による絞り込み
- 10件単位のページング

### API機能

- 認証不要の疎通確認
- Sanctum認証付き社員一覧
- Sanctum認証付き社員詳細
- 検索、部署絞り込み、1〜100件のページング

## 4. 実行環境の全体像

```text
Windows 11
├─ Docker Desktop
└─ WSL2 / Ubuntu
   └─ training-app-2026
      └─ Laravel Sail / Docker Compose
         ├─ laravel.test
         │  ├─ PHP 8.3
         │  ├─ Laravel 12
         │  ├─ Composer
         │  └─ Node.js / pnpm
         └─ mysql
            └─ MySQL 8.4
```

PCの8080番ポートをLaravelコンテナの80番へ、PCの3307番をMySQLコンテナの3306番へ接続します。

LaravelからMySQLへはDocker network内で接続するため、`.env` の接続先は `DB_HOST=mysql` です。

### Windowsでの実行端末

Laravel Sailの `vendor/bin/sail` はLinux用のシェルスクリプトです。Windowsでは、WSL2にUbuntuをインストールし、Ubuntuターミナルから実行します。Docker Desktopが作成する `docker-desktop` ディストリビューションだけでは、Sailの実行に必要な `/bin/bash` が存在しない場合があります。

プロジェクトを `C:\mywork\training-app-2026` に置いた場合、Ubuntuからは次のパスで参照します。

```bash
cd /mnt/c/mywork/training-app-2026
./vendor/bin/sail up -d
./vendor/bin/sail ps
./vendor/bin/sail stop
```

PowerShellから操作する場合はSailを直接実行せず、Docker Composeを使用します。

```powershell
cd C:\mywork\training-app-2026
docker compose up -d
docker compose ps
docker compose stop
```

Artisanコマンドも端末によって書き方が異なります。

```bash
# Ubuntu
./vendor/bin/sail artisan migrate
./vendor/bin/sail artisan test
```

```powershell
# PowerShell
docker compose exec laravel.test php artisan migrate
docker compose exec laravel.test php artisan test
```

PowerShellで `./vendor/bin/sail` を実行して、次のエラーが表示された場合は、Ubuntuが未導入または既定のWSLディストリビューションが `docker-desktop` になっている可能性があります。

```text
execvpe(/bin/bash) failed: No such file or directory
```

確認には次のコマンドを使用します。

```powershell
wsl --list --verbose
```

一覧にUbuntuがなければ、管理者PowerShellで `wsl --install -d Ubuntu` を実行し、Windows再起動後にUbuntuの初期設定を行います。

## 5. 設計方針

### 5-1. MVCを追いやすくする

研修用アプリとして、Route、Controller、Model、Viewの責任を追いやすい構成にしています。

```text
Browser
  ↓ HTTP Request
Route
  ↓
Form Request
  ↓ 検証済みデータ
Controller
  ↓
Eloquent Model
  ↓
MySQL
  ↓
Blade View または API Resource
  ↓ HTTP Response
Browser / API Client
```

### 5-2. 入力境界をForm Requestへ分離する

社員登録・更新では `EmployeeRequest`、一覧・API検索では `EmployeeIndexRequest` を使います。

- Controllerへ入る前に認証状態と入力値を確認する
- `$request->all()` を保存しない
- `$request->validated()` の値だけを使う
- WebとAPIで検索条件の検証を共通化する

### 5-3. WebとAPIの責任を分ける

- Web ControllerはBladeまたはredirectを返す
- API Controllerは`EmployeeResource`を返す
- APIは `/api/v1` でversioningする
- APIのルート名には `api.v1.` を付け、Webルート名との衝突を防ぐ

### 5-4. 削除は論理削除にする

`Employee` は `SoftDeletes` を使用します。DELETE操作ではレコードを物理削除せず、`deleted_at` を設定します。

通常のEloquent検索とルートモデルバインディングは、削除済み社員を自動的に除外します。

### 5-5. ログへ個人情報を残しすぎない

社員の登録・更新・削除ログには、社員IDと操作利用者IDだけを記録します。社員氏名、メール、入力値全体は記録しません。

## 6. データ設計

### users

Laravel標準の認証利用者です。

| 主なカラム | 用途 |
| --- | --- |
| `id` | 利用者ID |
| `name` | 利用者名 |
| `email` | ログインメールアドレス |
| `password` | hash化したパスワード |
| `remember_token` | ログイン保持 |

### employees

| カラム | 型・制約 | 用途 |
| --- | --- | --- |
| `id` | bigint、主キー | 社員ID |
| `employee_number` | varchar(20)、unique | 社員番号 |
| `name` | varchar(100) | 氏名 |
| `department` | varchar(100)、nullable、index | 部署 |
| `joined_at` | date、nullable | 入社日 |
| `email` | varchar(255)、nullable | 社員メール |
| `created_at` | timestamp | 登録日時 |
| `updated_at` | timestamp | 更新日時 |
| `deleted_at` | timestamp、nullable | 論理削除日時 |

### personal_access_tokens

SanctumのAPI tokenを管理します。平文tokenは発行時だけ表示され、DBにはhash化された値が保存されます。

### cache・jobs・sessions

Laravel標準のキャッシュ、Queue、Session用テーブルです。Sail環境ではSession、Cache、Queueにdatabase driverを使用します。

## 7. バリデーション

### 社員登録・更新

| 項目 | ルール |
| --- | --- |
| 社員番号 | 必須、文字列、重複禁止 |
| 氏名 | 必須、文字列 |
| 部署 | 任意 |
| 入社日 | 任意 |
| メール | 任意 |

更新時の社員番号unique検証は、更新対象自身のIDを除外します。
文字数、日付、メール形式の検証強化は、初級課題として未実装です。

### 一覧・API検索

| 項目 | ルール |
| --- | --- |
| `search` | 任意、文字列、100文字以内 |
| `department` | 任意、文字列、100文字以内 |
| `per_page` | 任意、整数、1〜100 |

Web一覧は10件固定です。APIは`per_page`で変更できます。

## 9. 主要処理フロー

### 社員登録

```text
POST /employees
  ↓ auth middleware
EmployeeRequest
  ↓ authorize + validate
EmployeeController@store
  ↓ Employee::create(validated data)
employeesテーブル
  ↓ employee.createdログ
社員一覧へredirect
```

### 社員検索

```text
GET /employees?search=山田&department=開発部
  ↓
EmployeeIndexRequest
  ↓
EmployeeController@index
  ↓ Employee::search()
  ↓ Employee::department()
  ↓ orderBy(employee_number)
  ↓ paginate(10)
employees/index.blade.php
```

### API社員一覧

```text
GET /api/v1/employees
  ↓ auth:sanctum
EmployeeIndexRequest
  ↓
Api/EmployeeController@index
  ↓ Eloquent検索・ページング
EmployeeResource::collection
  ↓
JSON Response
```

## 13. セキュリティ上の要点

- `.env`、API token、パスワードをGitへ登録しない
- Seederの初期パスワードはローカル研修環境だけで使用する
- Webの社員管理にはSession認証、社員APIにはSanctum認証を必須にする
- ログインと利用者登録は5回/分に制限する
- Controllerでは `$request->validated()` の値だけを保存する
- 社員の氏名、メールアドレス、入力内容全体をログへ出さない
- `phpinfo()`などの診断ページをpublicへ置かない
- `sail down -v`や`docker compose down -v`はDBデータを削除するため、実行前に対象環境を確認する

`main`では、文字数、日付、メール形式の強化バリデーションを初級課題として意図的に未実装にしています。これは研修用の開始状態であり、完成版として十分な入力防御を備えているという意味ではありません。

中級の住所検索では、利用する外部APIの規約、HTTPS、タイムアウト、接続失敗時の表示を確認します。APIキーが必要なサービスを採用する場合は、キーを`.env`で管理し、BladeやJavaScript、Gitへ直接記載しません。

上級課題では、次の点を追加で確認します。

- 部署マスター移行前に不整合データを調査し、バックアップと復旧方法を決める
- CSVのセル値を適切にエスケープし、保存先とダウンロード権限を限定する
- アップロード画像の拡張子だけを信用せず、MIME typeとファイルサイズを検証する
- 保存ファイル名を利用者入力のまま使用せず、社員ごとに保存先を分離する

現時点では社員単位のPolicyと管理者権限は未実装です。ログイン利用者は全員、全社員を登録・更新・削除できます。認可機能は今回の初級・中級・上級課題の対象外です。

## 14. 講師のレビュー観点

### 初級

- Carbonで入社日から現在までの経過年数を計算できているか
- 検索条件がURLパラメータとしてページ移動後も保持されるか
- 列ヘッダーで昇順・降順を切り替え、不正なソート列を受け付けないか
- Form Requestに文字数、日付、メール形式のルールを追加しているか
- 正常系だけでなく境界値と入力エラーをテストしているか

### 中級

- 住所検索用のRoute、Controller、Viewを新規作成しているか
- 社員新規登録画面の横から住所検索画面へ遷移できるか
- 標準ライブラリを使った最低限の郵便番号検証があるか
- 外部APIのリクエストとレスポンスを本人が説明できるか
- 検索結果と失敗メッセージを同じ画面で安全に表示できるか
- APIのタイムアウトや通信失敗で画面全体が500エラーにならないか

### 上級

#### 部署マスター化

- 正規化の目的と、部署マスターテーブルへ分離する理由を説明できるか
- migrationでテーブル作成、初期データ投入、社員テーブル変更を管理しているか
- 既存データの不整合を確認し、安全な移行順序と復旧方法を説明できるか
- 社員新規登録画面の部署をマスター由来のプルダウンに変更しているか
- 一覧画面が壊れないために必要な最低限の横断修正を確認したか

#### CSV出力

- 一覧画面の検索条件と同じ条件で社員を抽出しているか
- 社員番号の昇順、全カラムのヘッダー、Shift-JIS、CRLFを満たすか
- カンマ、ダブルクォート、改行を含む値をエスケープできているか
- `YYYYMMDD-hhmmss.csv`の名前でFlysystemの`storage/app/temp`へ保存しているか
- 実装コメントを本人が説明でき、SQLのWHERE句との対応を理解しているか

#### プロフィール画像

- 新規登録時に画像をFlysystem経由で保存しているか
- `img_id`を用いて`storage/app/employees/img/{img_id}/`配下へ分離しているか
- MIME typeとファイルサイズを検証しているか
- 社員名を押したモーダルで画像を横幅100%表示できるか
- 画像がない場合もエラーにせず、空のモーダルを表示できるか
- CSV用と画像用のfilesystem設定を分離し、設定競合を避けているか
