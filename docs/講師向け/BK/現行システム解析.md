# 現行 training-app システム解析

## 1. このアプリは何か

現行版は、新入社員が PHP と Laravel の基本を学ぶための社員管理アプリです。主題は、認証付き Web アプリにおける MVC、Eloquent、Blade、バリデーション、REST API、テストです。

利用者は登録・ログイン後、社員を一覧表示し、氏名または部署で検索し、登録・更新・論理削除できます。API は Sanctum 認証下で一覧と詳細を読み取れます。

## 2. 技術構成

| 領域 | 現行構成 | 根拠・補足 |
| --- | --- | --- |
| 言語 | PHP `^8.2` | README は PHP 8.3 を前提 |
| フレームワーク | Laravel `^11.31` | `composer.json` |
| Web 認証 | Laravel Breeze `^2.3` | 認証コントローラーと Blade を生成済み |
| API 認証 | Laravel Sanctum `^4.0` | `/api/user` と社員 API に適用 |
| UI | Blade + Bootstrap 5 + Alpine.js | 社員画面は Bootstrap のモーダルを利用 |
| データベース | MySQL 8.0 想定 | テストは SQLite in-memory |
| テスト | PHPUnit 11 / Laravel Testing | ソース上は計69テスト |
| 品質 | Pint、PHPStan、Larastan | 設定ファイルあり |
| 配備 | GitHub Actions → EC2/SSH | PR close を契機とした workflow |

重要な点として、`package-lock.json` はあるのに `package.json` がありません。この状態では Node/Vite 側の依存関係を再現できず、フロントエンドの正式なビルド手順が成立しません。社員画面は CDN 読み込みにも依存しており、Laravel 標準の Vite 構成と実装が混在しています。

## 3. 機能一覧

### Web

- `/` — Laravel の welcome 画面
- `/dashboard` — 認証・メール確認後、社員一覧へリダイレクト
- `/employees` — 一覧、氏名・部署の部分一致検索、10件単位のページング
- `/employees` `POST` — 社員登録
- `/employees/{employee}` `PUT/PATCH` — 社員更新
- `/employees/{employee}` `DELETE` — 論理削除
- `/profile` — プロフィール更新、パスワード更新、アカウント削除
- Breeze 標準の登録、ログイン、ログアウト、メール確認、パスワード再設定

### API

- `/api/ping` — 認証不要の疎通確認
- `/api/user` — Sanctum 認証ユーザー取得
- `/api/employees` — 認証付き一覧、検索、部署フィルター、ページング
- `/api/employees/{employee}` — 認証付き詳細

社員 API は読み取り専用です。Web と同じ CRUD API が存在するわけではありません。

## 4. データモデル

`employees` は次の単純な構造です。

| カラム | 型 | 必須 | 意味 |
| --- | --- | --- | --- |
| `id` | bigint | yes | 主キー |
| `name` | varchar(100) | yes | 氏名 |
| `department` | varchar(100) | no | 部署 |
| `joined_at` | date | no | 入社日 |
| `email` | varchar(100) | no | メールアドレス |
| `created_at`, `updated_at` | timestamp | no | 監査用時刻 |
| `deleted_at` | timestamp | no | 論理削除時刻 |

`Employee` モデルは一括代入対象を限定し、`joined_at` を日付へキャストします。また、氏名・部署検索と部署完全一致のローカルスコープを持ちます。社員とユーザーの関連、部署マスタ、権限、履歴テーブルはありません。

## 5. 処理の流れ

Web の社員操作は、ルート → `EmployeeController` → `Employee` → Blade という素直な MVC です。登録・更新時はコントローラー内で検証し、成功後に一覧へリダイレクトします。API は API コントローラーから `EmployeeResource` を通し、日付書式と勤続期間を加えた JSON を返します。

ログ処理は `EmployeeLogging` trait に集約されています。操作の理解には役立ちますが、更新前後の社員データ全体をログへ保存するため、メールアドレス等の個人情報がログに残ります。

## 6. テストの現状

テストコード上は69テストあります。内訳の中心は以下です。

- Employee モデル: fillable、日付 cast、Factory、論理削除・復元
- Employee Resource: JSON 形式、日付、勤続期間
- Web: 認証、一覧、検索、ページング、登録、検証、更新、削除、モーダル
- API: 認証、一覧、検索、部署絞り込み、ページング、詳細、404、論理削除除外
- Breeze: 登録、ログイン、メール確認、パスワード、プロフィール

ただし解析時点では `vendor` がないため、69件が現在すべて成功することは確認していません。「テストが存在する」と「テストが成功する」は分けて扱う必要があります。

## 7. 良い点

- 教材の主題が明確で、MVC の処理を追いやすい
- Web と API がモデルの検索スコープとログ処理を再利用している
- 論理削除、Factory、Seeder、Resource、認証まで学習範囲が広い
- 機能テストと単体テストが主要経路を広く覆う
- 既存文書に構成図、ER図、シーケンス、API、検証、テストの説明がある

## 8. 2026版で先に解消すべき問題

### 最優先: セキュリティと再現性

1. `.env` が Git 管理されています。アプリキーや接続情報は漏えい済みとして扱い、追跡解除だけでなく値のローテーションが必要です。
2. `public/phpinfo.php` が公開領域かつ Git 管理下にあります。本番環境情報を露出するため削除対象です。
3. 例外ログにスタックトレース、URL、IP、ユーザー情報を常時記録します。個人情報やクエリ文字列中の秘密情報が残る可能性があります。
4. 社員更新ログが更新前後の全データを記録し、社員メールアドレスも保存します。監査要件を決め、許可項目だけを記録すべきです。
5. `package.json` 欠落によりフロント依存を再構築できません。ロックファイルだけでは再現性を保証できません。

### 高優先: 入力・権限・API 境界

1. `EmployeeController` は `$request->all()` を渡しています。fillable による防御はありますが、`$request->validated()` を返す Form Request の方が入力境界を明示できます。
2. API の `per_page` は上限だけを `min()` で処理し、整数・正数であることを検証していません。0、負数、非数値への仕様を決める必要があります。
3. `joined_at` に未来日を許すか、社員メールを一意にするかが未定義です。
4. 認証はありますが社員操作に対する Policy/権限差はありません。ログインユーザー全員が全社員を変更・削除できます。
5. API にバージョン番号がなく、レスポンス契約を変更しにくい構成です。

### 高優先: CI/CD

1. workflow 名は自動配備ですが、テスト・静的解析・フォーマット検査を実行しません。
2. `pull_request: closed` で main 向け PR が閉じられるたびに動き、merge 済みかを判定していません。
3. `actions/checkout@v2` と `configure-aws-credentials@v1` は古いメジャー指定です。
4. SSH の host key 検証を無効化し、EC2 上で直接 `git pull` しています。
5. セキュリティグループを開いた後に処理が失敗すると、閉鎖ステップが実行されない可能性があります。
6. 配備先パスが `プロジェクトディレクトリ` のプレースホルダーのままです。

### 中優先: 設計と教材品質

1. Web の検証ルールが store/update で重複しています。
2. コントローラー、モデル、ルートに説明コメントが多すぎ、学習者が実コードの要点を見失いやすくなっています。
3. `years_of_service` は将来の入社日に対しても絶対差に近い表示になり得ます。業務ルールを先に決めるべきです。
4. 社員一覧の並び順が明示されていないため、DB によって表示順が安定しません。
5. 論理削除は実装されていますが、復元画面や復元 API はありません。
6. `storage:link` の説明はある一方、現行社員機能に画像アップロードはありません。
7. `docs` と `docks` が混在しています。2026版では正式名称をどちらかへ統一すべきです。

## 9. ドキュメントの信頼性

既存文書は量が多く、研修資料として有用です。ただし「実装を説明した仕様」と「将来案・一般論」の境界が曖昧です。たとえば API は読み取り専用なのに CRUD 全般を連想させる記述があり、セットアップには未実装の画像アップロードを前提とする説明があります。

2026版では各文書に「実装済み」「演習で追加」「参考」のラベルを付け、コードから確認できる事実を基準に更新するのが安全です。
