# 初級・中級・上級の社内研修アプリ設計

## 1. 研修目的

本研修では、Webアプリケーションにおけるフロントエンド、バックエンド、データベースの関係とMVCの役割を理解し、既存のLaravelアプリを調査しながら、基本的な機能追加・修正・テスト・報告を経験します。

期間内にすべてを自力で完成させることは必須としません。課題の完了数よりも、支援を受けながら開発工程を経験し、処理を説明できること、自分の到達点と次の課題を明らかにすることを重視します。

## 2. 振り返りから反映した設計原則
| 振り返り | 2026年の対応 |
| --- | --- |
| 初級・中級・上級で現在地が分かりやすかった | 各段階に目的、前提、必須、発展、完了確認を明記する |
| 説明会後の実装が有効だった | 各段階を説明→デモ→演習→レビュー→本人説明で進める |
| 動いたが理解していない箇所があった | コード提出だけでなく処理説明を必須にする |
| SQLがLaravelでの操作につながらなかった | 各課題にSQL・Eloquent対応確認を含める |
| Progateからプチアプリへの段差が大きかった | 初級前にコード追跡と1項目追加の小課題を置く |
| 日報の形式が欲しかった | 共通の日報テンプレートを用意する |
| 回答者と説明方針に偏りがあった | 主担当・副担当・共通レビュー表を課題ごとに定める |
| フロント、バック、DB、クラウドの関係が難しかった | 同じ社員管理アプリを段階的に育て、つながりを保つ |

## 3. 共通アプリの考え方

題材は現行版と同じ「社員管理」とします。レベルごとに別の題材へ切り替えると、業務理解と環境構築を毎回やり直すことになります。同じアプリを次の順に育てることで、前段の知識が次段でどう使われるかを見せます。

```text
初級: 1テーブルの社員CRUDを読み、小さく変更する
  ↓
中級: 部署マスタとのリレーション、検索、検証、テストを加える
  ↓
上級: 権限、API、監査、CIを加え、業務運用を考える
```

標準環境は参照タスクの決定に合わせ、PHP 8.3、Laravel 12、Bootstrap 5とします。DotinstallのLaravel 11は基本概念を学ぶ補助教材とし、実装、回答、レビュー、成果物はLaravel 12を正とします。

## 4. レベルの全体像

| 観点 | 初級 | 中級 | 上級 |
| --- | --- | --- | --- |
| 主目的 | MVCとDBの流れを追う | 複数テーブルと品質を扱う | 安全な業務システムとして設計する |
| 作業 | 手順・ヒント付き改修 | 仕様から実装 | 選択肢を比較して設計・実装 |
| DB | `employees` 1テーブル | `departments` と外部キー | 権限・監査を含む複数モデル |
| Web | 一覧、登録、編集、削除 | 検索、部署管理、ページング | 認可、復元、例外設計 |
| API | 読み取り・観察 | API Resourceと検証 | versioning、token権限、契約 |
| テスト | 既存テストを読み1件追加 | Feature Testを仕様から追加 | 認可・API・失敗系を含む品質ゲート |
| 成果説明 | 変更したファイルを説明 | 設計とテスト根拠を説明 | 判断、リスク、運用を説明 |

レベルは人の優劣ではなく、課題の段階です。全員が初級から開始し、進捗に応じて中級・上級へ進みます。

## 5. 研修の基本サイクル

各単元は次のサイクルで進めます。

1. 基礎説明 — 用語と今回使う理由を確認する
2. 講師デモ — 画面操作からコード、DBまでを追う
3. ミニ課題 — 変更範囲が分かる課題を実施する
4. 個人実装 — Issue単位で実装する
5. レビュー — 共通観点でコードと理解を確認する
6. 本人説明 — どこをなぜ変更したかを説明する
7. 振り返り — 日報に理解、疑問、支援内容を残す

初級では説明とデモを厚くし、中級ではヒントとレビュー中心、上級では設計相談とレビュー中心へ移します。

## 6. 到達度の記録

課題ごとに、完了・未完了だけでなく支援レベルを記録します。

| 段階 | 定義 |
| --- | --- |
| A | 自力で実装し、理由も説明できた |
| B | ヒントがあれば実装し、説明できた |
| C | 手順説明を受けて実装し、流れを説明できた |
| D | 一部未完成だが、調査内容と詰まった点を説明できた |

Dは失敗ではありません。どの知識を補えば次へ進めるかが明確なら研修成果として扱います。

## 7. 共通成果物

各段階で次を更新します。

- 動作するLaravelアプリまたは作業途中のブランチ
- GitHub IssueとPull Request
- 画面・Route・Controller・Model・Tableの対応表
- DB定義またはER図
- テスト結果
- README
- 本人の振り返り

コードだけを成果物にしないことで、「コピーして動いた」と「理解して変更した」を区別します。

## 8. Git運用

- 1課題を1 Issueにする
- 1 Issueを原則1ブランチ、1 Pull Requestにする
- PR本文に変更理由、確認方法、未解決事項を書く
- 講師は答えを直接書く前に、調査箇所または小さなヒントを提示する
- レビュー完了後、本人が指摘内容を説明してからマージする

推奨タグは `level:beginner`、`level:intermediate`、`level:advanced`、`blocked`、`question` です。

## 9. 日報テンプレート

```markdown
## 今日実施したこと

## できるようになったこと

## 分からなかったこと・困っていること

## 自分で調べたこと

## 受けたヒント・支援

## 明日実施すること

## 講師に確認したいこと

理解度: 1 / 2 / 3 / 4 / 5
進捗: 予定より遅い / 予定どおり / 予定より早い
```

## 10. 講師運営

- 各単元に主担当と副担当を置く
- 技術方針の最終判断者を決める
- 週1回、講師だけで質問、進捗、レビュー基準を共有する
- 朝会は15分以内とし、進捗、困り事、今日の作業だけを確認する
- 技術相談は担当者を割り当て、回答が一部の講師へ集中しないようにする
- 個人差が大きい場合、予定を圧縮せず必須課題と発展課題を分ける

## 12. 完成の定義

研修アプリの準備完了は、完成コードがあるだけでは不十分です。starterから各課題を実施でき、課題文、期待動作、テスト、レビュー観点、ヒント、模範解答が対応し、別の講師が同じ基準で運営できる状態を完成とします。
